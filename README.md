# RustProbe 设计大纲

## 1. 项目目标

RustProbe 是一个面向 **非 Root Android 设备** 的安全流量分析探针 App，目标是在 **不依赖 Root** 的前提下，尽最大限度捕获设备网络流量，并基于 Rust 完成高性能解析、关联、检测与展示。

当前阶段优先级如下：

1. 先做到无 Root 条件下的最大可见性。
2. 再做到稳定的协议解析、应用归因与按应用筛选。
3. 再做恶意外联、挖矿行为与资源负载检测。
4. 最后完善告警、取证导出与 UI 体验。

## 2. 现实边界

无 Root 环境下，不可能像系统级探针那样直接抓所有网卡原始报文。首版方案必须接受以下事实：

- 主路径只能基于 `VPNService` 接管经过系统 VPN 栈的 IP 流量。
- 能稳定看到的是“经过本探针虚拟网卡的流量”，不是底层物理网卡原始帧。
- 通常拿不到链路层信息，比如 Ethernet 头、ARP、真实网卡统计。
- 在 `VPNService` + TUN 模式下，通常无法稳定拿到远端链路层 `MAC` 地址。
- 通常看不到未经过 VPNService 的极少数系统内部通信。
- App 自身必须避免把自己的回环代理流量再次捕获，防止死循环。

因此项目目标应该表述为：

**在非 Root 条件下，以 `VPNService` 为核心，尽量覆盖设备上绝大多数应用层外联流量，并完成可归因、可分析、可导出。**

同时，项目的主要分析目标应明确聚焦于：

- 单个或多个指定应用的流量监听与行为分析
- 恶意域名 / 恶意 IP 外联检测
- 可疑挖矿行为识别
- 应用资源负载分析，包括 CPU 与内存占用趋势
- 域名、URL、IP、`MAC`、端口等网络对象的展示、分类与聚合分析

## 3. 核心设计原则

### 3.1 无 Root 优先

项目默认运行在标准 Android 用户环境中，不依赖：

- Root
- Magisk
- 内核模块
- `AF_PACKET` 原始套接字
- `iptables` / `nftables`

关键能力围绕 Android 官方允许的能力建设，尤其是：

- `VPNService`
- 应用 UID / 包名映射
- 本地代理转发
- 用户态 Rust 分析引擎

### 3.2 捕获优先于解密

第一阶段先解决“尽量看得见”问题，不把 HTTPS 明文还原当成前置目标。默认重点分析：

- 五元组
- 时序
- DNS
- TLS ClientHello / SNI / ALPN / JA3 类指纹材料
- HTTP 明文流量
- 证书元数据

后续如需做 MITM 解密，应单独设计证书注入、用户授权与兼容性策略，不与基础探针强耦合。

### 3.3 用户态主导

优先使用用户态 Rust 完成大部分工作。首版主线为：

- `VPNService` 提供 TUN 流量入口
- Rust 从 TUN 读取 IP 数据包
- `pnet` 做 IPv4 / IPv6 / TCP / UDP 解析
- `tokio` 做异步管线
- `rayon` 做 CPU 密集分析并行

### 3.4 分层解耦

采集、转发、重组、协议解析、归因、检测、存储、展示分层，避免后续替换 UI、转发器或解密模块时牵动全局。

### 3.5 应用级分析优先

这个项目不是通用抓包器，而是偏安全分析探针。因此首版设计应优先保证：

- 可以只监听单个应用
- 可以同时监听多个应用
- 可以快速切换目标应用集合
- 所有分析结果都能回溯到具体应用

应用对象建议统一使用以下标识：

- UID
- package name
- app label

### 3.6 对象级观测优先

除了应用维度，探针还应围绕“网络对象”建立统一观测模型。

首版重点对象：

- 域名
- URL
- IP
- 端口
- `MAC`

说明：

- 域名、URL、IP、端口应作为一等分析对象参与展示、搜索、分类、聚合和告警。
- `MAC` 需要区分可见性来源；无 Root + TUN 模式下通常无法稳定获得远端 `MAC`，但本机接口信息、网关信息或少量可推断场景可作为补充元数据展示。

## 4. 总体架构

建议采用 “Android 容器层 + Rust 核心层 + 前端展示层” 三层结构。

### 4.0 项目整体架构图

```text
+----------------------------------------------------------------------------------+
|                                 RustProbe App                                    |
+----------------------------------------------------------------------------------+
|                            Presentation / UI Layer                               |
|----------------------------------------------------------------------------------|
| Dashboard | 应用监控 | 对象分析 | 连接列表 | 会话详情 | 告警中心 | 设置/导出     |
+-------------------------------------------+--------------------------------------+
                                            |
                                            | WebView / Tauri-style Frontend
                                            v
+----------------------------------------------------------------------------------+
|                          Android Shell / Control Layer                           |
|----------------------------------------------------------------------------------|
| MainActivity | App Selector | Config Manager | Export Manager | Permission Flow  |
| VPNService   | protect()    | TUN fd bridge  | IPC bridge     | Lifecycle Mgmt   |
+-------------------------------------------+--------------------------------------+
                                            |
                                            | JNI / UDS / Local WS
                                            v
+----------------------------------------------------------------------------------+
|                             Rust Core / Analysis Layer                           |
|----------------------------------------------------------------------------------|
| Capture  | Relay  | Parse  | Flow  | Attribution | Object Agg | Metrics         |
| Detect   | Alert  | Store  | IPC   | Rule Engine | Threat Intel | Export         |
+-------------------------------------------+--------------------------------------+
                                            |
                                            | read/write packets
                                            v
+----------------------------------------------------------------------------------+
|                             Android VPN / Network Layer                          |
|----------------------------------------------------------------------------------|
| VPNService TUN | TCP/UDP relay sockets | DNS/HTTP/TLS/QUIC traffic               |
| Target Apps    | Allowed/Disallowed Apps | IPv4 / IPv6 external connections       |
+----------------------------------------------------------------------------------+
```

说明：

- 上层是展示和控制，下层是抓包与分析，最底层是 Android 无 Root 能力边界。
- Android 层负责 `VPNService`、权限、应用选择和生命周期，Rust 层负责真正的数据通路与安全分析。
- UI 不直接依赖内部流表或检测器，只通过 IPC 消费摘要结果、对象聚合结果和告警事件。

### 4.1 Android 容器层

职责：

- 提供 App 外壳
- 启动和管理 `VPNService`
- 管理权限申请与用户授权
- 管理日志目录、配置目录、导出目录
- 提供系统事件桥接
- 启动 WebView / Tauri 风格 UI

建议：

- Android 宿主使用 Kotlin
- `VPNService` 放在 Android 侧实现
- Rust 核心以 `cdylib` 或 JNI 绑定方式集成

### 4.2 Rust 核心层

职责：

- 从 TUN 读取数据包
- IP 层解包
- 流表管理
- 协议识别
- 连接归因
- 资源负载采集
- 规则检测
- 数据缓存与导出

建议拆成多个 crate：

- `rustprobe-capture`: TUN 读写、转发调度、会话入口
- `rustprobe-parse`: 基于 `pnet` 的 L3/L4 解析
- `rustprobe-flow`: 流表、会话重组、超时清理
- `rustprobe-attrib`: UID / 包名 / 应用信息归因
- `rustprobe-detect`: 规则、特征、告警
- `rustprobe-metrics`: CPU、内存、连接数、带宽等负载指标
- `rustprobe-store`: 内存缓存、文件落盘、索引
- `rustprobe-ipc`: 与 Android / UI 通信

### 4.3 前端展示层

职责：

- 实时流量列表
- 会话详情
- 域名 / 证书 / DNS / SNI 展示
- 单应用 / 多应用对比视图
- CPU / 内存 / 带宽负载视图
- 告警面板
- 搜索过滤与导出

建议：

- UI 技术栈从简，前端可用 Vue 或 React
- Android 首版优先考虑 Kotlin WebView 承载本地前端
- 如果后续 Tauri for Android 适配顺利，再替换容器

说明：在 Android 端，Tauri 可以继续保留为方向，但它不应成为首版关键路径。先把探针和数据通路跑通，比 UI 容器选型更重要。

### 4.4 Actor 拓扑图

核心并发模型建议采用 “Actor + Pipeline” 组合：Actor 负责状态隔离与消息传递，Pipeline 负责数据处理顺序。

首版推荐拓扑如下：

```text
+-------------------+        +------------------+
| Android App/UI    | <----> | UiGatewayActor   |
| Kotlin + WebView  |        | 状态推送/命令入口 |
+---------+---------+        +--------+---------+
          |                           |
          | JNI / UDS / WS            |
          v                           v
+-------------------+        +------------------+
| VpnService        | -----> | SessionCtlActor  |
| TUN fd / protect  |        | 配置/启停/模式切换|
+---------+---------+        +--------+---------+
          |                           |
          | fd pass                   |
          v                           |
+-------------------+                 |
| TunCaptureActor   | ----------------+
| TUN读包/写包      |
+---------+---------+
          |
          | PacketEvent
          v
+-------------------+
| RelayActor        |
| TCP/UDP relay     |
| NAT/protect socket|
+---------+---------+
          |
          | ParsedPacket / RelayEvent
          v
+-------------------+      +-------------------+
| FlowActor         | ---> | AttributionActor  |
| 流表/会话生命周期 |      | UID/包名归因      |
+---------+---------+      +---------+---------+
          |                          |
          | FlowEvent                | AppFlowEvent
          +-------------+------------+
                        |
                        v
              +-------------------+
              | MetricsActor      |
              | CPU/内存/带宽聚合 |
              +---------+---------+
                        |
                        | AnalysisEvent
                        v
              +-------------------+
              | DetectionActor    |
              | 恶意外联/挖矿检测 |

                   |         |
         AlertEvent|         |RecordEvent
                   v         v
          +----------------+ +----------------+
          | AlertActor     | | StorageActor   |
          | 告警聚合/抑制  | | 落盘/导出/索引 |
          +--------+-------+ +--------+-------+
                   |                  |
                   +--------+---------+
                            |
                            v
                   +------------------+
                   | UiGatewayActor   |
                   | 列表/图表/告警推送|
                   +------------------+
```

设计要点：

- `TunCaptureActor`、`RelayActor`、`FlowActor` 是主数据通路，优先保证稳定和低延迟。
- `AttributionActor`、`MetricsActor`、`DetectionActor` 是分析通路，允许适度异步和降级。
- `AlertActor` 与 `StorageActor` 独立，避免告警聚合或落盘阻塞主链路。
- `UiGatewayActor` 只接收摘要事件，不直接读取内部共享状态。
- `SessionCtlActor` 统一处理模式切换，例如全局监听、单应用监听、多应用监听。

### 4.5 Actor 职责边界

建议首版 Actor 边界如下：

- `SessionCtlActor`: 管理运行状态、目标应用集合、配置热更新、启动与停止。
- `TunCaptureActor`: 负责 TUN fd 读写、基础报文读取、入口统计。
- `RelayActor`: 负责 TCP/UDP relay、socket 生命周期、`protect()` 调用、NAT 映射。
- `FlowActor`: 负责五元组流表、会话超时、状态汇总。
- `AttributionActor`: 负责 UID、包名、应用标签归因与缓存。
- `MetricsActor`: 负责每应用连接数、带宽、CPU、内存、趋势聚合。
- `DetectionActor`: 负责恶意域名/IP、矿池特征、异常行为规则检测。
- `AlertActor`: 负责告警抑制、去重、分级、合并。
- `StorageActor`: 负责 JSONL、SQLite、重建 PCAP、导出任务。
- `UiGatewayActor`: 负责把内部事件整理成 UI 可消费的流式数据。

## 5. 流量捕获方案

这是项目成败的核心。

### 5.1 首选方案：VPNService + TUN 全量接管

无 Root 条件下，首选且几乎唯一可行的系统级方案是：

- 通过 `VPNService` 创建 TUN 虚拟网卡
- 把设备应用流量路由到该 TUN
- 在用户态读取原始 IP 包
- 在本地完成分析与转发

这条路径的优点：

- 无 Root 可用
- 能覆盖大多数应用外联流量
- 可以拿到 IPv4 / IPv6 数据包
- 可以在转发前完成审计、标记、记录

这条路径的代价：

- 需要自己实现或集成用户态转发器
- 看到的是 IP 包，不是底层链路帧
- 本地 DNS、代理、回环场景需要额外处理
- 性能和兼容性比系统原生网络路径更敏感

### 5.2 转发模式

探针不能只“读包不转发”，必须提供完整的数据转发闭环。推荐采用：

1. TUN 读入 IP 包
2. Rust 解析五元组
3. 建立用户态连接映射
4. 通过受保护 socket 向真实目标转发
5. 响应流量再封装回 TUN

实现上建议分两类：

- TCP：使用用户态 TCP relay / stream proxy
- UDP：使用用户态 UDP relay

要点：

- 所有外连 socket 必须通过 `VpnService.protect()` 排除出 VPN，避免流量回灌
- 需要维护 NAT 映射表
- 需要处理超时、重传、半关闭和异常中断

### 5.3 关于 `pnet` 的使用位置

在无 Root 架构下，`pnet` 不再负责物理网卡抓包，而是负责解析从 TUN 读出的 IP 包：

- IPv4 / IPv6
- TCP / UDP / ICMP
- payload 切片

这仍然是合理的，因为项目的分析核心依旧围绕报文级结构化，只是输入源从原始网卡变成了 TUN。

### 5.4 应用归因方案

无 Root 环境下，归因路径会比 Root 更受 Android 能力边界约束。建议采用多种方式组合：

1. Android `ConnectivityManager` / `VpnService` 可获得的网络上下文
2. `PackageManager` 提供 UID 到包名映射
3. `getConnectionOwnerUid()` 一类系统能力
4. 会话创建时缓存 UID 到五元组关联
5. 前台应用状态、已安装应用信息作为补充标签

归因目标建议先定为：

- UID
- package name
- app label

不要把首版目标定得过高到 PID / 进程名级别，因为无 Root 下这部分稳定性较差。

### 5.5 单应用与多应用监听

探针需要支持两种工作模式：

1. 全局监听模式
2. 目标应用监听模式

目标应用监听模式下，用户可以：

- 选择单个应用进行重点分析
- 选择多个应用组成观察集合
- 按 UID / 包名维护监听白名单或黑名单

实现建议：

- 优先利用 `VPNService` 的 `addAllowedApplication()` / `addDisallowedApplication()` 做应用级纳管
- 当平台兼容性或业务模式不适合只纳管目标应用时，TUN 侧仍接收全局流量
- 归因后在分析层按目标应用集合做过滤、聚合和告警
- UI 允许在“全局视角”和“目标应用视角”之间切换

原因：

- Android 无 Root 下不适合依赖底层定向抓包
- `VPNService` 原生支持按应用选择接管范围，适合单应用或多应用重点分析
- 先支持“按应用纳管”和“全局接入后筛选”两种模式，工程上最稳

### 5.6 特殊流量与盲区

首版必须明确记录以下场景的处理策略：

- IPv6：必须支持
- QUIC / HTTP3：必须识别并分类
- DoH / DoT：必须尽量识别
- 本地代理流量：必须避免死循环
- 分应用 VPN 排除：必须支持
- `127.0.0.1` / `::1`：通常难以像 Root 抓包那样完整观测

项目文档中应明确写出：

**无 Root 首版追求的是“覆盖大多数真实外联流量”，不是“百分之百拿到所有内核可见包”。**

## 6. 数据处理流水线

建议采用多阶段流水线：

1. TUN 采集层
2. 解包层
3. 流表层
4. 协议识别层
5. 归因层
6. 指标层
7. 检测层
8. 存储与展示层

这张图描述的是首版最核心的动态路径：

- 前半段解决“如何在无 Root 下接入流量并保持网络可用”。
- 中间段解决“如何把包变成会话、对象和应用画像”。
- 后半段解决“如何形成检测、告警、导出和界面展示”。

### 6.1 TUN 采集层

- Kotlin 创建并配置 TUN
- 将文件描述符传给 Rust
- Rust 持续读取 IP 包
- 只做最小必要元信息提取
- 快速投递到下游 channel

`tokio` 适合承担：

- TUN I/O 事件循环
- 转发 socket 管理
- IPC
- UI 推送
- 定时任务

### 6.2 解包层

使用 `pnet` 解析：

- IPv4 / IPv6
- TCP / UDP / ICMP
- 基础 payload 切片

这里不做太重的逻辑，只完成结构化。

### 6.3 流表层

维护：

- 五元组索引
- 正反向字节计数
- 首包时间 / 末包时间
- TCP 状态摘要
- NAT / relay 映射
- 异常标记

必要时支持：

- TCP 分段重组
- UDP 会话聚合
- QUIC 初始包缓存

### 6.4 协议识别层

先做轻量协议识别：

- DNS
- TLS ClientHello / ServerHello
- HTTP/1.x
- HTTP/2 连接识别
- QUIC Initial 基础字段

输出内容包括：

- 域名
- URL
- SNI
- ALPN
- User-Agent
- Host
- 远端 IP
- 源 / 目的端口
- 证书摘要
- JA3 / JA3S 类指纹

### 6.5 对象聚合层

对象聚合层负责把原始会话和协议元数据整理成面向分析的对象视图。

首版建议支持以下对象类型：

- `DomainObject`
- `UrlObject`
- `IpObject`
- `PortObject`
- `MacObject`

首版建议支持以下聚合维度：

- 按应用聚合域名
- 按应用聚合 URL
- 按应用聚合远端 IP
- 按应用聚合端口
- 按风险等级聚合对象
- 按时间窗口聚合对象访问频次
- 按协议聚合对象分布

每类对象建议至少统计：

- 首次出现时间
- 最近出现时间
- 命中次数
- 关联应用数
- 关联会话数
- 上下行字节
- 风险标签

对象层的价值：

- 让 UI 不只停留在“连接列表”
- 让恶意域名 / IP / URL / 端口的聚合分析更直接
- 让导出结果更适合取证与研判

### 6.6 指标层

指标层负责把“流量行为”和“资源占用”结合起来，形成按应用的综合画像。

首版建议采集以下指标：

- 每应用连接数
- 每应用上行 / 下行带宽
- 每应用 DNS 请求频率
- 每应用活跃会话数
- 每应用 CPU 占用
- 每应用内存占用

建议输出两类数据：

- 实时快照
- 时间序列趋势

说明：

- CPU / 内存指标优先做应用级统计，不强求线程级精度
- 如 Android 公开 API 无法稳定拿到所有细粒度数据，则至少保证前台可见应用和已授权场景下的数据可用

### 6.7 检测层

首版先做规则型检测，不上复杂模型：

- 非白名单域名访问
- 明文传输敏感信息
- 异常 DNS 高频请求
- 可疑外联 IP
- 可疑端口扫描或心跳
- 自签名 / 异常证书
- 后台高频上传

首版需要重点补充两类安全检测：

- 恶意外联检测
- 挖矿行为检测

CPU 密集型检测放到 `rayon`：

- 指纹计算
- 批量规则匹配
- 大量会话聚合分析

### 6.8 恶意外联检测

重点关注：

- 命中威胁情报的域名 / IP
- 低信誉 ASN / 国家或地区外联
- 异常高频 DNS 解析
- 短周期轮询 C2 风格连接
- SNI / Host 与证书主体明显不一致
- 大量失败连接与快速切换目标 IP

建议告警维度：

- 应用
- 域名 / IP
- URL
- 端口
- 首次发现时间
- 命中规则
- 风险等级

### 6.9 挖矿行为检测

挖矿类恶意行为通常同时具备网络与资源两侧特征，首版可以做规则和画像结合检测。

重点关注：

- 长时间连接矿池域名或矿池 IP
- 与 `stratum` 类协议特征相似的 TCP 流量
- 持续稳定的小包心跳与提交模式
- 后台长连接且上行周期性提交
- CPU 长时间异常偏高
- 网络活动与资源占用同步升高

首版策略建议：

- 维护矿池域名 / IP 特征表
- 识别常见矿池端口与协议握手特征
- 结合 CPU、内存、连接持续时间做综合评分
- 将“疑似挖矿”定义为中高风险事件，而非直接定性

## 7. Android 集成设计

### 7.1 进程模型

建议分成两个逻辑部分：

- App 主进程：负责 UI、配置、任务控制
- VPNService + Rust 核心：负责抓包、转发与分析

如果后续稳定性要求更高，可以把 VPNService 放入独立进程。

### 7.2 启动流程

1. App 启动
2. 展示 VPN 授权说明
3. 用户授权 `VPNService`
4. Android 创建 TUN 接口
5. 将 TUN fd 交给 Rust
6. Rust 开始采集、转发和分析
7. UI 通过 IPC 拉取状态和实时事件

### 7.3 IPC 方式

可选方案：

1. JNI 直接调用
2. Unix Domain Socket
3. 本地 TCP `127.0.0.1`

建议首版：

- 控制面用 JNI 或 UDS
- 实时事件用 WebSocket / 流式 JSON

原因是调试成本低，UI 更换成本也低。

## 8. 数据存储与导出

首版不建议上重数据库，优先轻量可控：

- 会话级导出

建议导出能力：

- 会话列表导出

## 10. 性能设计

### 10.1 并发模型

- 核心并发模型采用 `Actor + Channel`
- `tokio` 管 TUN I/O、socket I/O、actor 调度与异步消息传递
- `rayon` 管计算密集分析
- 采集与分析之间用有界队列削峰

首版建议：

- 主链路 actor 常驻运行
- actor 之间通过 `tokio::mpsc` 传递事件
- 重计算任务由 `DetectionActor` 或 `MetricsActor` 投递到 `rayon` 执行
- 尽量避免多个 actor 共享同一份可变状态

### 10.2 关键优化点

- TUN 读写缓冲池复用
- 避免在热路径做字符串分配
- 流对象复用
- 批量提交 UI 更新
- 将重解析与重计算放到后台

### 10.3 背压与降级

必须设计高负载降级策略：

- 高负载时暂停深度解析，只保留元数据
- 降低 UI 推送频率
- 大流量会话做摘要化
- 限制单连接缓存深度
- 记录每一级降级事件

## 11. 安全设计预留

## 15. 当前建议结论

如果目标是尽快做出一个能打的无 Root 原型，最稳的路线是：

1. Android 宿主先用 Kotlin。
2. 核心探针全用 Rust。
3. 抓包主路径走 `VPNService` + TUN。
4. `pnet` 负责解析 TUN 中的 IP 包，而不是物理网卡抓包。
5. 并发模型用 `tokio` + `rayon`。
6. 分析模型围绕“按应用观察 + 恶意外联 + 挖矿行为 + 资源负载”展开。
7. UI 先追求“能看、能筛、能导出”，不要过早卷复杂交互。
8. Tauri 可以继续观察，但首版不要把它当关键路径依赖。

一句话总结：**无 Root 方案的重点不是“抓所有底层包”，而是把 `VPNService` 路线做到稳定，并把结果沉淀为按应用的安全画像，重点发现恶意外联和疑似挖矿行为。**
