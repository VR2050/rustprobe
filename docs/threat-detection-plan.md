# RustProbe 威胁检测改造计划

本文档用于承接当前这一轮“分析精度 + 系统态仪表盘 + 应用流量分析”完成后的下一阶段规划。

接下来项目重心不再只是“看见流量”，而是逐步升级成：

- 能稳定提取威胁线索
- 能把线索关联到应用、域名、IP、端口、协议与时间窗口
- 能形成结构化告警，而不是只展示原始观测结果

## 1. 当前基础能力

当前已经具备较好的检测前置条件：

- `VPNService -> TUN -> Rust` 主链路稳定
- Rust 侧已有 `flow / object / analytics` 三类结构化输出
- 应用归因链路已接入，并在继续提准
- 域名线索来源已覆盖：
  - DNS
  - TLS SNI
  - QUIC Initial SNI
  - HTTP Host
- 前后端都已经开始做统计口径修正，而不是只做展示

这意味着后续可以从“采集器”正式转向“检测器”。

## 2. 下一阶段总目标

下一阶段建议聚焦 5 条主线：

1. 把检测输入口径继续打准
2. 把检测模型从单点规则升级成多线索关联
3. 建立稳定的威胁情报与本地规则体系
4. 让告警可解释、可追溯、可复盘
5. 为后续实时告警 UI / 历史检索 / 风险评分铺结构

## 3. 核心设计方向

### 3.1 从“流量统计”走向“威胁证据”

后续检测不要只看单一指标，例如：

- 某个域名访问次数很多
- 某个端口流量较大
- 某个应用网速突然升高

这些都可以作为信号，但不应该直接等同于威胁。

建议把检测单位改成 `Evidence -> Correlation -> Alert` 三层：

1. `Evidence`
   - 单条证据，例如：
   - 命中可疑域名
   - 命中非常见端口
   - TLS SNI 与 DNS 不一致
   - 某应用短时高频连接大量目的 IP
   - QUIC / DoH / DoT 隐匿特征
2. `Correlation`
   - 把同一时间窗口、同一应用、同一对象的多条证据聚合
3. `Alert`
   - 输出最终告警，包含风险等级、原因、命中证据、关联流与对象

### 3.2 从“单规则命中”走向“风险评分”

建议后续每个应用 / 域名 / IP / flow / 时间窗口都支持累计风险分。

例如：

- 可疑域名命中：`+40`
- DoH/DoT 隐匿：`+15`
- 夜间异常活跃：`+10`
- 高温 + 高放电 + 高联网活跃：`+10`
- 同时访问多 ASN / 多国家 IP：`+15`

当总分超过阈值时再触发正式告警，而不是看到一条信号就报警。

这样能显著降低误报率。

## 4. 优先级路线图

### P0：检测输入继续提准

目标：先把“送进检测器的数据”做得更可信。

任务：

- 统一流量字节口径：
  - 区分链路字节
  - 传输层 payload 字节
  - 对象统计字节
- 统一域名归因优先级：
  - `TLS SNI / QUIC SNI > HTTP Host > DNS`
- 增加时间窗口一致性约束：
  - `active_flows`
  - `top objects`
  - `burst`
  - `trend`
- 对 `unattributed` 流量单独建检测维度，不混入普通 app
- 增加方向信息：
  - 上行 / 下行
  - 首包时间
  - 结束时间
  - 持续时长

验收：

- 相同流量在 flow / object / analytics 三层统计结果相互可对齐
- 同一域名不会因来源抖动而频繁改变归因
- 时间窗口指标可复现、可解释

### P1：建立第一版本地威胁规则引擎

目标：先有一套不依赖外部在线服务、可以离线运行的检测规则。

建议先做 4 类规则：

1. `IOC 命中规则`
   - 恶意域名
   - 恶意 IP
   - 高风险 ASN
   - 高风险端口
2. `隐匿通信规则`
   - DoH
   - DoT
   - QUIC-only 高隐匿访问
   - TLS SNI 缺失但持续外连
3. `行为异常规则`
   - 短时间连接大量不同 IP
   - 单应用对大量域名高频探测
   - 小包高频外连
   - 非常规时间高活跃
4. `资源-流量联动规则`
   - 高放电 + 高联网
   - 高温 + 高联网
   - 系统态异常与某应用流量峰值同步

验收：

- 能在本地输出结构化 `AlertEvent`
- 每条告警至少包含：
  - 风险等级
  - 标题
  - 摘要
  - 命中证据
  - 关联应用
  - 关联对象
  - 时间窗口

### P2：做关联分析，不再只报单点命中

目标：减少误报，提升“像安全分析而不是日志高亮”。

任务：

- 为 `app / domain / ip / window` 引入关联缓存
- 建立 `EvidenceRecord`
- 同一实体在窗口内累计多个命中后，再升级成告警
- 引入抑制逻辑：
  - 同一对象不要每秒重复报警
  - 同一应用同类风险合并
- 引入衰减逻辑：
  - 风险分会随时间衰减

验收：

- 相同异常不会短时间刷屏
- 告警总数下降，但有效告警占比上升

### P3：威胁情报与可维护规则体系

目标：让规则不再散落在代码里。

任务：

- 定义规则配置格式：
  - JSON
  - TOML
  - YAML 三选一，建议 TOML
- 支持本地威胁情报文件：
  - domain blocklist
  - ip blocklist
  - allowlist
  - risk overrides
- 支持规则版本号与更新时间
- 支持“规则命中原因”反查

验收：

- 不改代码也能更新基础 IOC
- 告警能反查是哪个规则命中的

### P4：为检测 UI 与复盘准备数据结构

目标：让后续前端做“安全工作台”而不是“文本输出页”。

建议新增结构：

- `EvidenceRecord`
- `AlertRecord v2`
- `RiskScoreSnapshot`
- `ThreatTimelineSnapshot`

前端后续可直接展示：

- 告警时间线
- 按应用风险排行
- 按域名/IP 风险排行
- 某应用威胁画像
- 某告警证据链

验收：

- UI 不需要重新拼接日志字符串
- 历史复盘能看到“为什么报警”

## 5. 具体模块改造建议

### 5.1 `rustprobe-parse`

目标：继续提高检测线索密度。

优先方向：

- 更稳的 HTTP 请求元数据
- 更稳的 TLS ClientHello 解析
- QUIC Initial 线索补强
- DNS 响应侧信息

建议新增：

- `user_agent` 最小提取
- `http method`
- `content-type` 线索
- TLS 版本与 cipher suite 摘要
- DNS 响应 IP / CNAME 关联

### 5.2 `rustprobe-flow`

目标：让 flow 成为检测主载体。

建议新增字段：

- `bytes_up`
- `bytes_down`
- `duration_ms`
- `last_payload_direction`
- `first_domain_source`
- `resolved_ips`

并继续强化：

- 元数据可信度优先级
- 同一 flow 多来源线索合并
- 对象更新去重

### 5.3 `rustprobe-metrics`

目标：让分析模块从“看板统计”升级成“检测输入层”。

建议新增：

- `burst score`
- `fanout score`
- `beacon score`
- `encrypted-only ratio`
- `domain churn`
- `destination diversity`

### 5.4 `rustprobe-detect`

目标：变成真正的规则引擎，而不是单条 CPU 示例。

建议拆成：

- `ioc_rules`
- `behavior_rules`
- `resource_rules`
- `correlation_engine`
- `suppression_engine`

## 6. 推荐的开发顺序

建议按下面顺序推进，不要一开始就并行铺太大：

1. 先完成 `P0`
   - 检测输入口径继续提准
2. 再完成 `P1`
   - IOC + 隐匿通信 + 基础异常行为
3. 然后做 `P2`
   - 关联与抑制
4. 最后推进 `P3 / P4`
   - 规则配置化
   - 告警可解释化
   - UI 友好输出

## 7. 下一轮建议完成线

如果下一轮能做到下面这些，就算方向非常正确：

- `rustprobe-detect` 不再只是 CPU 示例，而是至少有 5-10 条真实规则
- `AlertRecord` 能附带命中证据
- `analytics` 支持检测所需的 burst / fanout / diversity 指标
- 本地情报文件可以加载
- 前端能看到第一版“威胁告警列表”

到那一步，RustProbe 就会从“流量分析工具”进一步升级成“本地可运行的移动侧网络威胁检测原型”。

## 8. 暂不建议立即做的事

以下方向重要，但建议排在上述主线之后：

- HTTPS MITM 解密
- 云端大模型判恶
- 远端情报实时拉取
- 大规模历史索引数据库
- 复杂可视化安全图谱
- 自动拦截与主动处置

原因不是它们不重要，而是：

- 当前最需要的是先把检测输入与本地规则引擎打实
- 只有证据链可靠，后续拦截和自动化才不会误伤
